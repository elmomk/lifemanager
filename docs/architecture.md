# Life Manager — Architecture Guide

## Overview

Life Manager is a mobile-first, offline-capable Progressive Web App (PWA) built with **Rust + Dioxus 0.7** (fullstack, compiled to WebAssembly) and styled with **Tailwind CSS v4**. It provides five personal-management modules:

| Module | Purpose |
|--------|---------|
| **To-Dos** | General task tracking with dates |
| **Groceries** | Shopping list with quick-add chips |
| **Shopee Pick-ups** | Convenience-store package tracking (codes, stores) |
| **Watchlist** | Movie/Series/Anime/Cartoon tracker |
| **Cycle Tracker** | Menstrual cycle logging with predictions |

---

## Tech Stack

| Layer | Technology |
|-------|-----------|
| Language | Rust |
| Frontend framework | Dioxus 0.7 (compiled to Wasm) |
| Server framework | Dioxus fullstack (Axum 0.8 under the hood) |
| Styling | Tailwind CSS v4 |
| Server database | SQLite via `rusqlite` + `r2d2` connection pool |
| Authentication | Tailscale user header (`Tailscale-User-Login`) |
| Deployment | Nginx reverse proxy → `dx serve`, Tailscale serve |

## How the Pieces Fit Together

### 1. Fullstack in One Binary

Dioxus 0.7 fullstack compiles the same Rust codebase twice:

- **`target_arch = "wasm32"`** — the client bundle (runs in the browser)
- **Native target** — the server binary (runs on the host, serves the Wasm bundle + handles API calls)

Code that should only exist on one side is gated:

```rust
// Server-only (SQLite, file I/O, etc.)
#[cfg(not(target_arch = "wasm32"))]

// Client-only (IndexedDB via rexie, DOM APIs)
#[cfg(target_arch = "wasm32")]
```

### 2. Routing

`src/route.rs` defines the `Route` enum consumed by Dioxus Router:

```
/           → redirects to /todos
/todos      → Todos page
/groceries  → Groceries page
/shopee     → Shopee page
/watchlist  → Watchlist page
/period     → Cycle Tracker page
```

All routes are wrapped by `AppLayout`, which provides the fixed header (with theme toggle) and the bottom `TabBar`.

### 3. Data Flow (Request Lifecycle)

```
┌─────────────────────────────────────────────────┐
│  Browser (Wasm)                                 │
│                                                 │
│  Page mounts                                    │
│    └─► use_effect calls server function          │
│         └─► HTTP POST to /_server_fn/...        │
│              └─► Server deserializes request     │
│                   └─► auth::user_from_headers()  │
│                   └─► db::pool().get() → SQLite  │
│                   └─► query/insert/update/delete │
│                   └─► serialize response         │
│              └─► Wasm receives Vec<Model>        │
│         └─► Signal updated, UI re-renders       │
└─────────────────────────────────────────────────┘
```

Each page follows this pattern:

1. **Mount**: `use_effect` fires an async fetch via a `#[server]` function
2. **User action** (add/toggle/delete): spawns an async server call, then bumps a `refresh` signal
3. **Refresh signal change**: `use_effect` dependency triggers, items reload

### 4. Server Functions

Server functions use the `#[server(FnName)]` attribute macro. At compile time, Dioxus generates:
- A client-side stub that serializes args and sends an HTTP request
- A server-side handler that deserializes, runs the function body, and returns the result

Example:

```rust
#[server(ListChecklist)]
pub async fn list_checklist(category: ItemCategory) -> Result<Vec<ChecklistItem>, ServerFnError> {
    let headers: axum::http::HeaderMap = extract().await?;
    let user_id = user_from_headers(&headers);
    let conn = pool().get()?;
    // ... SQLite query ...
    Ok(items)
}
```

### 5. Database Schema

SQLite (`life_manager.db`) with four tables, all scoped by `user_id`:

| Table | Key Columns |
|-------|------------|
| `checklist_items` | id, user_id, text, date, done, category (`todo`/`grocery`), created_at |
| `shopee_packages` | id, user_id, title, store, code, picked_up, created_at |
| `watch_items` | id, user_id, text, media_type, done, created_at |
| `cycles` | id, user_id, start_date, end_date, symptoms (JSON array) |

Connection pooling: `r2d2::Pool<SqliteConnectionManager>` stored in a `OnceLock` singleton, initialized once at startup.

### 6. Authentication

`src/server/auth.rs` reads the `Tailscale-User-Login` header injected by Tailscale serve. Falls back to `"local"` in development. Every API function extracts headers and calls `user_from_headers()` to scope data per user.

### 7. UI Component Architecture

```
AppLayout
├── Header (title + theme toggle)
├── Outlet (page content)
│   └── [Page Component]
│       ├── QuickAddChips (preset items)
│       ├── Add Form (inputs + button)
│       └── Item List
│           └── SwipeItem (per item)
│               ├── Left swipe → Delete (red)
│               └── Right swipe → Toggle done (green)
└── TabBar (bottom navigation)
```

**SwipeItem** handles touch gestures with direction locking — it detects whether the user is scrolling vertically or swiping horizontally, preventing conflicts. Thresholds: ±120px to trigger an action.

**QuickAddChips** renders a horizontally-scrollable row of preset items (e.g., "Milk", "Eggs") that populate the add form on tap.

### 8. Styling & Design Language

- **Glassmorphism**: semi-transparent backgrounds (`bg-white/70`, `bg-gray-800/70`) with `backdrop-blur`
- **Rounded corners**: `rounded-2xl` for cards, `rounded-3xl` for containers
- **Dark/light mode**: system-aware via Tailwind `dark:` variants, with a manual toggle persisted in `localStorage`
- **Module colors**: Blue (todos), Orange (groceries), Blue (shopee), Purple (watchlist), Pink (cycle)
- **Mobile-first**: `max-w-lg` (32rem) centered layout

### 9. PWA & Offline Support

- **Service worker** (`assets/sw.js`): cache-first strategy for GET requests. On install, skips waiting; on activate, cleans old caches and claims clients.
- **Manifest** (`assets/manifest.json`): `display: standalone`, themed blue/dark, scoped to `/lifemanager/`.
- **Registration** (`assets/sw-register.js`): auto-registers the service worker on page load.

### 10. Deployment Pipeline

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Tailscale   │     │    Nginx     │     │  dx serve    │
│  serve :443  │────▶│   :7000      │────▶│   :8080      │
│  /lifemanager│     │  proxy_pass  │     │  fullstack   │
└──────────────┘     └──────────────┘     └──────────────┘
```

- **dx serve** runs on port 8080, serving both the Wasm frontend and the API
- **Nginx** on port 7000 reverse-proxies to 8080, handling the `/lifemanager` base path
- **Tailscale serve** exposes port 7000 over HTTPS on the tailnet at `/lifemanager`
